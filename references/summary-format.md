# 论文总结统一规范

本文档同时承担两件事：

1. 作为阅读论文时的分析框架，确保关键信息不遗漏。
2. 作为最终输出时的写作规范，确保结果简洁、清楚、可复用。

默认要求是：

- 思考时按完整框架分析。
- 输出时按压缩格式呈现。
- 除非用户明确要求长格式，否则不要默认写成冗长的 10 节报告。
- 默认中文输出必须参考 `references/summary-example-zh.md` 的版式、段落组织、信息密度和表达风格；除非用户明确要求其他格式，否则应尽量贴近该示例。

## 一、任务目标

当用户提供本地 PDF 路径、arXiv 链接或其他论文链接时，你需要：

1. 下载或定位论文文件。
2. 认真阅读论文的摘要、方法、实验与结论部分；必要时查看引言、附录和元数据。
3. 核验论文的发表状态；如果尚未正式发表，则给出最早公开时间。
4. 先完成结构化分析，再输出简洁中文总结。
5. 将结果写入本地 `summary.md`，并在需要时同步写入 Notion。

## 二、分析框架

阅读和整理时，至少覆盖以下信息：

### 1. 基本信息

- 论文标题
- 作者
- ArXiv ID；若没有则写 `N/A`
- 最早公开时间
- 是否已发表；若已发表，写明会议、期刊或状态
- 主要研究领域或类别
- 阅读依据：`source` 或 `pdf`

### 2. 研究目标

回答以下问题：

- 这篇论文想解决什么问题
- 为什么这个问题重要
- 现有方法的主要缺陷是什么

### 3. 方法

需要提炼而不是照抄。重点识别：

- 方法的核心思想
- 整体流程或系统结构
- 与已有方法相比最关键的设计变化
- 训练、推理、检索、数据构造或奖励设计中的关键机制

### 4. 评测与结果

至少回答：

- 在什么数据集、任务或环境上评测
- 主要对比基线是谁
- 最能代表论文贡献的结果数字是什么
- 优势体现在哪些维度
- 是否有代价、限制或负面结果

### 5. 局限与风险

至少考虑：

- 方法成立所依赖的前提是否过强
- 实验覆盖是否不足
- 成本、复杂度、稳定性、泛化性是否存在问题
- 论文是否有作者未充分展开但实际重要的风险点

## 三、核验要求

输出前尽量完成以下核对：

1. 核验发表状态与最早公开时间。
2. 通读 PDF 的摘要、方法和实验部分，记录关键流程与代表性数值。
3. 如果存在会议页面、期刊页面或 arXiv 页面，尽量交叉核对状态和关键信息。
4. 对结果数字优先保留最有代表性的 1 到 3 个，不要堆砌表格。

若某项信息无法确认，应明确写“不确定”或“未在文中明确说明”，不要猜测。

## 四、默认输出格式

默认输出应尽可能简短，但不能损失核心信息。结构如下：

整体版式必须参考 `references/summary-example-zh.md`。也就是说，最终结果应更接近“面向人阅读的整理稿”，而不是僵硬模板。

如果本规范的抽象描述与实际排版选择之间存在歧义，默认以 `references/summary-example-zh.md` 作为中文输出样式的最终参照。

### 1. 标题行

格式建议：

```markdown
1. 论文标题（发表状态或最早公开时间）
```

例如：

```markdown
1. Self-Distilled Agentic Reinforcement Learning（arXiv，2026-05-15）
```

如果用户不需要编号，也可以省略最前面的 `1.`。但默认可保留这种“整理条目”风格。

如果论文有代码仓库、项目页或作者明确给出的实现链接，可在标题后补充：

```markdown
代码: ...
```

如果没有，就不要硬补。

### 2. 一句话总结

用通俗、信息密度高的一句话说明：

- 论文提出了什么
- 核心方法是什么
- 相比什么更好
- 解决了什么关键问题

要求：

- 不说空话
- 不写成摘要翻译腔
- 让没读论文的人也能快速明白
- 默认直接接在标题和补充信息之后，形成一段自然导语

可参考 `summary-example-zh.md` 中“作者提出……从而……”的写法。

### 3. 五段式主体

使用下面五个部分：

- `**研究动机**`
- `**创新点**`
- `**方法**`
- `**结果**`
- `缺点`

要求：

- `研究动机`、`创新点`、`方法`、`结果` 这四个标签必须加粗。
- 不使用有序列表。
- `缺点` 可以不加粗，以贴近中文 example 的现有风格；如果统一加粗也可以，但默认沿用 example。
- 可以使用短段落或无序要点。
- 若内容较多，允许在每个部分内部使用无序列表；不要使用 `1. 2. 3.` 这类有序列表。
- 每一部分尽量压缩到“够用而不啰嗦”。
- 方法部分必须凝练、易懂，不要陷入公式复述。
- 结果部分只保留最关键的实验结论和数字。
- 缺点部分应尽量真实，不要只写客套话。
- 若某篇论文特别适合“整理者 / 代码 / 项目页 / 数据集链接”这类前置信息，可以像 `summary-example-zh.md` 那样放在一句话总结之前或之后，但不要为了凑格式强行添加无关字段。

## 五、写作风格要求

- 全文默认用中文写。
- 语言要自然，避免生硬翻译腔。
- 优先解释“这篇论文到底做了什么”，而不是机械复述章节名。
- 尽量具体，不要泛泛而谈。
- 不要把论文所有实验都搬出来，只保留最有代表性的内容。
- 不要使用脚本、正则或模板自动拼接正文。
- 所有判断必须以论文原文、元数据或可核验来源为依据。

## 六、长格式的使用条件

只有在用户明确要求详细版、结构化归档版或 10 节完整分析版时，才使用更长的格式。

长格式可覆盖以下维度：

1. 论文快照
2. 研究目标
3. 方法概览
4. 数据与评测
5. 关键结果
6. 优势
7. 局限与风险
8. 可复现性
9. 实践启示
10. 简短结论

但这不是默认输出方式。

## 七、最小元数据块

即使最终输出采用简洁风格，也应在内部分析阶段保留以下字段：

```markdown
- ArXiv ID: ...
- Title: ...
- Authors: ...
- Publish date: ...
- Primary category: ...
- Reading basis: source/pdf (...)
```

如果不是 arXiv 论文，可写：

```markdown
- ArXiv ID: N/A
```

## 八、一个简单模板

```markdown
1. 论文标题（发表状态或最早公开时间）

代码: ...

一句话总结：...

**研究动机**
...

**创新点**
...

**方法**
...

**结果**
...

缺点
...
```

更接近默认风格的模板如下：

```markdown
1. 论文标题（发表状态或最早公开时间）
代码: ...（如有）

作者提出……，通过……，从而……，相较于……，在……任务上……

**研究动机**
- ...
- ...

**创新点**
- ...
- ...

**方法**
- ...
- ...

**结果**
- ...
- ...

缺点
- ...
- ...
```

若不确定最终排版，直接模仿 `references/summary-example-zh.md` 的中文风格、段落组织和信息密度。

## 九、执行原则

- 先分析完整，再压缩输出。
- 先保证准确，再追求简洁。
- 先抓住论文真正贡献，再写表面结构。
